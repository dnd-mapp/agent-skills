# Review gate before GitHub provisioning in configure-issue-tracker

When `configure-issue-tracker` runs with a GitHub Tracker Backend, all GitHub API writes (creating labels, Project fields, org-level issue types, and optionally creating the Project itself) are deferred until after the full interview completes. The skill presents a summary of all decisions made and all GitHub changes that will be applied, and requires explicit user approval before any write is executed. The alternative — incremental provisioning mid-interview — was rejected because a failed API call partway through leaves the repo in a partially configured state that is hard to reason about or roll back. Batching writes at the end also ensures the user has seen the full picture before anything touches shared GitHub infrastructure.
